## Problem Statement

Existing Python-based MCP agent frameworks (Hermes, Openclaw, plus the broader LangChain /
LangGraph ecosystem) have three production pain points:

1. **Networking overhead** — Python's async story is decent but the GIL and per-call serialization
   tax make parallel tool fan-out expensive at scale.
2. **Tool calling latency** — Reflection-based tool dispatch, slow JSON marshaling, and process
   model (uvicorn workers vs. true concurrency) compound.
3. **Deployment ergonomics** — Multi-file Python apps with venvs, lockfiles, and binary deps
   are painful to ship to edge / appliance / sidecar contexts.

**Tenant** is a Go-native MCP agent framework that solves these by leaning on Go's strengths
(goroutines, single-binary deploys, low GC overhead, strong typing for tool schemas) while
preserving the modular plugin model that makes Python frameworks pleasant to extend.

## What Makes This Cool

A **single 20MB binary** that you `scp` to a box and it serves an MCP-native agent with
iMessage, GSuite, X, SQL, web research, a personal-wiki memory layer, and online learning
from interactions. No venv. No Docker required. Sub-100ms tool dispatch overhead. Deterministic
goroutine fan-out for parallel tool calls. That's the headline.

The deeper "whoa" is the **wiki-as-memory** + **MDP-shaped action history**: every interaction
appends to a transparent, inspectable knowledge graph (Karpathy-style: plain files, no magic),
and the framework learns which tool sequences worked for which user intents over time.

## Constraints

- **Language:** Go ≥ 1.22 (generics, range-over-func)
- **Runtime target:** Linux/amd64 + Linux/arm64 + Darwin/arm64 (Mac is required for iMessage —
  see Premise 3)
- **Deployment:** single static binary preferred; optional sidecar container for sandboxed tools
- **LLM provider:** model-agnostic via MCP, but defaults to Anthropic Claude (since that's the
  user's stack — see Nebula CLAUDE.md)
- **License posture:** TBD — assumption is Apache-2.0 for max adoption. **FLAG: confirm.**

## Premises

**P1: Go gives meaningful wins over Python for an agent framework runtime.** — **CONFIRMED, with caveat.**
The wins are real for: parallel tool fan-out (goroutines vs. asyncio), cold-start (no
interpreter), memory footprint (~10x smaller), and deployment. The wins are *negligible* for
single-tool-call latency, because >95% of wall-clock time is the LLM HTTP call itself. Go is
the right choice if your agent makes 5+ parallel tool calls per turn or you're shipping to
constrained environments. It's overkill if you make one tool call per turn on a beefy server.

**P2: "Methodical modular approach ensures stability and bandwidth."** — **CHALLENGED.**
"Bandwidth" is an unusual word here. I'm reading it as *throughput under concurrent load*. If
that's right, the lever isn't modularity per se — it's **backpressure, connection pooling, and
bounded queues** at every plugin boundary. Modularity is the means; the actual stability
properties come from explicit resource budgets per plugin. **FLAG: confirm "bandwidth" interpretation.**

**P3: iMessage capability.** — **CONFIRMED but ARCHITECTURALLY CONSTRAINING.**
There is no official iMessage API. Real options:
- Read `~/Library/Messages/chat.db` (SQLite) on macOS + send via AppleScript / JXA / `osascript`
- Pay for BlueBubbles / sendblue / AirMessage (~$5-25/mo per number, third-party trust)
- Use macOS Shortcuts + URL schemes (limited)

This forces a hard architectural fork: **(a) the host process runs on Mac**, or **(b) iMessage
lives in a separate Mac-side sidecar daemon that the main agent talks to over gRPC / NATS**.
Recommend (b) — keeps the core framework portable, isolates the Mac dependency.
**Risk:** Apple changes the chat.db schema or sandboxing rules ~yearly. Plan for breakage.

**P4: GSuite + Twitter/X integrations are commodity.** — **CONFIRMED.**
- GSuite: `google.golang.org/api` (official, mature) + OAuth2 device flow
- Twitter/X: Official v2 API, Basic tier ~$200/mo or Free tier (very limited writes).
  **FLAG: cost matters here. If this is for personal use, free tier may not be enough.**

**P5: Deep research via web tools + reasoning.** — **CONFIRMED, but scope this tightly.**
"Deep research" is a Pandora's box (think OpenAI Deep Research: 5-30 minute multi-step runs).
For v1, scope to: tool-augmented single-shot research (web search + fetch + summarize) with
optional iterative refinement loop bounded by token/time budgets. **Don't** try to compete with
OpenAI/Anthropic's hosted Deep Research on day one.

**P6: SQL interfacing.** — **CONFIRMED. Decision: SQLite + Postgres for v1.** Use `database/sql`
+ driver registry (`modernc.org/sqlite` for pure-Go SQLite, `jackc/pgx` for Postgres). MySQL,
MSSQL, etc. deferred to v1.1+. Safety guards: no DDL by default, per-query row budget, EXPLAIN
on slow queries.

**P7: "Karpathy-style Wiki Connector."** — **CHALLENGED for ambiguity, then CONFIRMED.**
I'm reading this as: a transparent, plain-files, minimal knowledge layer modeled on Karpathy's
pedagogical style (nanoGPT / micrograd — single file, no framework magic, fully readable).
Concretely: index a directory of markdown files (Obsidian / personal wiki / git-backed notes)
with embeddings + BM25, expose as MCP resource + tool. No "RAG framework" overkill.
**FLAG: confirm this interpretation. If you mean "wiki as agent memory" vs "wiki as RAG source,"
the design differs.**

**P8: MDP + RL for agent self-improvement.** — **STRONGLY CHALLENGED for v1.**
The honest reality: real online RL during agent inference is research-grade and has not shipped
in production agents anywhere. The practical version of this for a real product is:
1. **Log every (prompt, tool sequence, outcome, user reaction) tuple** — easy, do this day 1
2. **Retrieve similar past traces as in-context examples** for new prompts — easy, high ROI
3. **Identify high-success patterns and promote them to "skills"** (named tool sequences) — medium
4. **Offline RLHF / DPO on collected traces** to fine-tune a *router* model that picks tools — hard
5. **True online RL with a critic network updating during inference** — research-grade, skip v1

What's reachable in v1: items 1-3. Items 4-5 become possible *because* you logged the data well
in v1. **Recommend: ship #1-3 as "Continual Learning" and reserve "MDP/RL" language for the
v2 milestone when you have real data.**

**P9: "Self-improvement skills to quietly enhance the product per use case."** — **CONFIRMED but
needs guardrails.** This is exciting and dangerous. Auto-modifying behavior in production
without user awareness is how products silently degrade. Design: every "learned skill" is
visible in a `tenant skills list` CLI, can be disabled per-tenant, and shows the trace evidence
that promoted it. Transparent self-improvement, not opaque drift.

## Approaches Considered

### Approach A: **Wrapper-First — Stand on mcp-go**

Build Tenant as an *agent layer* on top of `mark3labs/mcp-go`. Don't reimplement MCP itself.
Focus on agent orchestration, planning loop, integrations, and learning layer.

| | |
|---|---|
| Summary | Use mcp-go for protocol, build agent runtime + integrations + learning layer on top |
| Effort | M (2-3 months solo, evenings + weekends) |
| Risk | Low |
| Pros | Fastest to working demo. Reuses battle-tested protocol code. Easy to upstream improvements. Compatible with the broader MCP server ecosystem out of the box. |
| Cons | Inherits mcp-go's design choices (some are not ideal for high-throughput). Limits how deep optimization can go. The "rebuild for efficiency" pitch is weakened if you're a wrapper. |
| Reuses | mcp-go (protocol), database/sql, google.golang.org/api, official x/oauth2, gjson |

### Approach B: **Ground-Up Rewrite — Full MCP Stack + Agent Layer**

Reimplement the MCP protocol (JSON-RPC over stdio + SSE + WebSocket), the agent runtime,
plugin model, and integrations. Aim for the cleanest possible architecture with
performance-first defaults (zero-allocation hot paths, pooled buffers, structured concurrency
via `errgroup` + cancellation propagation).

| | |
|---|---|
| Summary | Greenfield, opinionated, performance-first; nothing inherited |
| Effort | XL (6-12 months solo) |
| Risk | High |
| Pros | Maximum architectural control. Strongest "efficiency" story. Cleanest abstractions. Can design plugin model around stability/backpressure from line one. |
| Cons | You're rebuilding plumbing the community already maintains. Months before you ship anything interesting. Integration with the MCP server ecosystem requires conformance testing. Likely never reaches feature parity with Python frameworks. |
| Reuses | Standard library, golang.org/x/sync, a few well-chosen deps (gjson, sqlite) |

### Approach C: **Lateral — Mac-First, Personal-Use, "OS-Resident Agent"** ⚡

Reframe entirely. Don't build a general framework. Build **a single user's agent that lives on
their Mac**, with iMessage + GSuite + X + wiki memory as native first-class plugins, and the
framework abstraction is *implied* (it's what falls out of building the thing). Ship the
working personal agent in 6 weeks. Extract framework boundaries in month 3 once you know which
abstractions are actually load-bearing.

| | |
|---|---|
| Summary | Build the user, not the framework. Framework emerges from a real working product. |
| Effort | S to start (6 weeks for working personal agent), M total |
| Risk | Low for product, Med for the framework story |
| Pros | Ships fast. iMessage requirement stops being an architectural constraint (host IS a Mac). You build framework abstractions only for problems you actually hit. Karpathy-style — start narrow and concrete. Highest learning rate. |
| Cons | "Tenant" name implies multi-tenancy / framework — branding mismatch. If goal is to build the framework as the product, this path delays that. Risk of over-fitting to one user's workflow. |
| Reuses | mcp-go (small uses), AppleScript bindings, google.golang.org/api |

## Recommended Approach

**Approach A (Wrapper-First) with explicit Approach C sequencing.**

Concretely: use mcp-go for protocol, build Tenant as the agent layer + integrations + learning
layer on top, but **build for yourself first**. The first user is you. The first integrations
that ship are the ones you'll use this week (iMessage, GSuite, wiki). Framework abstractions
get extracted from working code, not designed up front.

Why not pure C (Mac-first personal agent only): keeps "Tenant" branded as a framework so future
multi-tenant ambitions stay plausible. Pure C delays the framework story indefinitely.

Why not pure B (ground-up rewrite): 6-12 months to a working demo is too long; you'll lose
interest or the ecosystem will move. The "efficiency rebuild" pitch reads good but the actual
perf wins that matter (parallel tool fan-out, single-binary deploy, low memory) are achievable
on top of mcp-go. You don't need to own the protocol layer to get those.

### Plugin interface (the load-bearing abstraction)

Every plugin — local or remote — speaks MCP. Plugins register one of two ways:

```go
// In-proc: zero IPC overhead, plugin lives in the same binary
type Plugin interface {
    Name() string
    Tools() []mcp.Tool                      // tools this plugin exposes
    Resources() []mcp.Resource              // resources (e.g. wiki docs)
    Call(ctx context.Context, name string, args json.RawMessage) (mcp.Result, error)
}
tenant.Register(NewGSuitePlugin(creds))     // in-proc registration

// Subprocess: isolation, restart-on-crash, language-agnostic
tenant.Spawn("tenant-imessage-bridge", "--socket=/tmp/imsg.sock")  // stdio MCP server
```

Default to in-proc for trusted first-party plugins (GSuite, X, SQL, Wiki, Web Research) —
they're Go code we control. Use subprocess for plugins that need isolation (iMessage sidecar,
because it lives on a different machine) or untrusted third-party MCP servers.

**Key structural choices:**

- **Plugin interface** = MCP server (everything is an MCP server, even local plugins). See the
  Go interface sketch above. In-proc by default, subprocess for isolation.
- **iMessage sidecar** = separate binary that runs on a Mac, talks gRPC to the main agent over
  a unix socket (when co-located) or mTLS (when remote — cert generated at first run via a
  shared CA in `~/.config/tenant/ca/`). Lets the core stay portable (Linux deployment) while
  iMessage stays optional.
- **Wiki connector** = a goroutine that watches a directory (fsnotify), indexes markdown to
  SQLite-fts5 for keyword search + an embedding column for semantic search. Default embedding
  provider: **Voyage AI** (`voyage-3` — Anthropic's recommended partner, ~$0.06/M tokens).
  Fallback: local `nomic-embed-text` via Ollama for fully-offline mode. Exposes both as MCP
  resources and tools.
- **Continual learning (v1)** = append-only event log (`tenant.log` — JSONL) of every
  (prompt, tool sequence, outcome, latency, user reaction). Background indexer maintains a
  SQLite-fts5 keyword index + embedding column on the prompts. On each new prompt, planner
  retrieves top-K similar past traces by cosine similarity (K=5) and includes successful ones
  as in-context examples. **Success signal:** explicit `tenant ack` / `tenant undo` CLI
  commands (and chat-channel equivalents), OR absence-of-correction within 3 turns (heuristic,
  decays over time).
- **Skill promotion (deferred to v1.1)** = periodic job that scans the event log for repeated
  successful tool sequences, names them, registers as `tenant skills`. Each skill visible in
  CLI and toggleable per-tenant. **Not in v1 scope** — log + retrieve is enough to ship.

6. **Hosting** — your machine, a VPS, or the existing Nebula appliance as a runtime target?

## Distribution Plan

- **Binary releases:** GitHub Releases via `goreleaser`, signed for macOS (notarized) and Linux.
- **Install:** `curl -fsSL tenant.dev/install.sh | sh` (Homebrew formula in month 2).
- **Mac sidecar:** separate `tenant-imessage-bridge` binary, also via Releases. Documented as
  optional.
- **Config:** TOML at `~/.config/tenant/config.toml`. Secrets in OS keychain (macOS Keychain,
  Linux libsecret). NEVER in config file.
- **Telemetry:** opt-in, off by default. If on: anonymous skill-usage counts only, no prompts /
  responses.
- **CI/CD:** GitHub Actions matrix (linux/amd64, linux/arm64, darwin/arm64). Goreleaser handles
  multi-arch builds. Tag → release pipeline.

## Dependencies / Risks

- **iMessage breakage risk:** Apple changes chat.db schema or sandboxing → sidecar breaks.
  Mitigation: schema-version detection + fallback to AppleScript-only send mode.
- **iMessage operational gotchas:**
  - **Full Disk Access required** — sidecar binary must be granted FDA in System Settings →
    Privacy & Security. First-run installer should detect missing FDA and surface a clear
    instruction modal (no silent failure).
  - **chat.db is WAL-locked while Messages.app runs** — reader cannot use the live file safely.
    Sidecar strategy: copy `chat.db` + `chat.db-wal` + `chat.db-shm` to a temp dir before each
    read (cheap — file is small), or use SQLite's WAL-readonly open pattern with
    `?mode=ro&_busy_timeout=200`. Polling cadence: 1-2s with debounce.
  - **AppleScript send** has a ~250ms cold start. Acceptable for chat-paced interactions.
- **mcp-go API churn:** MCP spec is young; expect breaking changes. Pin to a known-good version
  and gate upgrades behind a compat test suite.
- **LLM cost runaway:** an agent in a planning loop with no guardrails can burn $100 in an
  evening. Budget per session, hard cap per day, alerts.
- **X API access:** if you don't pay, write quotas may make X integration a read-only / very
  rate-limited feature.

## What I noticed about how you think

- You said "**methodical modular approach to ensure stability and bandwidth during production
  runtime**" — that's not how someone hacking on a side project talks. That's someone who has
  watched production agents fall over and wants to architect for the failure mode they've seen.
  The instinct is right; the open question is whether you've named what's actually unstable.
- You called it "Tenant" — that's a load-bearing word. Multi-tenancy isn't an afterthought you
  bolt on. If the name is aspirational, fine; if it's the actual product thesis, the v1 scope
  needs to change a lot.
- You listed "Markov Decision Processes" and "Reinforced learning" as line items next to
  iMessage and SQL — those are *very* different categories of difficulty. Treating them as
  peers suggests you're more excited about the research angle than the integrations. That's
  worth noticing — research-grade items can quietly become the project's center of gravity
  while the boring integrations (which are actually what makes the product useful) get
  underbuilt. Watch for this.
- "Self-improvement skills to quietly enhance the product per use case" — the word **quietly**
  is doing a lot of work. Quiet self-improvement is how products silently degrade. Loud,
  visible self-improvement (where the user can see which skills were learned and turn them off)
  is how trustworthy products improve. Worth deciding which one you want.
